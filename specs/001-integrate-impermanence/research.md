# Research: Impermanence-Backed Workstation Persistence

## Decision: Use `nix-community/impermanence` as a flake input

The project will add `impermanence.url = "github:nix-community/impermanence"` to
`flake.nix`, make its `nixpkgs` input follow this repo's `nixpkgs`, and import
`inputs.impermanence.nixosModules.impermanence` through shared workstation
NixOS wiring.

Rationale: FlakeHub reports `nix-community/impermanence` as the current public
flake for impermanence, latest version `0.1.203`, revision
`7b1d382faf603b6d264f58627330f9faa5cba149`. The upstream flake is purpose-built
for persistent state declarations on ephemeral NixOS roots.

Sources:

- https://flakehub.com/flake/nix-community/impermanence
- https://github.com/nix-community/impermanence

Alternatives considered:

- Vendor the module source locally: rejected because the upstream flake is small
  and vendoring would create update drift.
- Reimplement persistence bind/link behavior: rejected because impermanence
  already provides the desired NixOS and Home Manager option model.

## Decision: Import NixOS impermanence only; do not manually import Home Manager output

Home Manager safe-haven declarations will be enabled only when the NixOS
impermanence module and NixOS Home Manager integration make the Home Manager
impermanence options available. The plan will not add an impermanence Home
Manager module directly to the standalone `homeConfigurations` module list.

Rationale: The upstream impermanence flake exposes a NixOS module as the primary
entrypoint. Its current flake declares Home Manager module outputs as deprecated
and asserts when they are imported manually. Keeping the import on the NixOS
side also matches this repository's ownership split: NixOS owns storage and
mount availability, Home Manager owns user path declarations when embedded.

Alternatives considered:

- Add the Home Manager module directly to `home-manager.sharedModules`: rejected
  because current upstream marks that path as deprecated.
- Use only NixOS `environment.persistence.users.<name>` declarations for user
  paths: viable, but less aligned with this repository's Home Manager ownership
  of user-session configuration.
- Enable safe-haven declarations for standalone Home Manager unconditionally:
  rejected because standalone profiles do not own host storage support.

## Decision: Keep dataset creation in disko and root wiping in ZFS boot logic

`workstation-disko.nix` remains the source of truth for datasets. The
impermanence module declares what persists; a repository-owned NixOS storage
module verifies prerequisites and rolls back `zroot/encrypted/ephemeral/root` to
`root@blank` during early boot.

Rationale: The existing shared disko layout already creates an encrypted ZFS
tree, an ephemeral root dataset mounted at `/`, and a `root@blank` snapshot via
`postCreateHook`. Upstream impermanence documentation treats deleting or
resetting root as a separate system setup concern and the module as the
persistence declaration layer.

Alternatives considered:

- Use tmpfs root: rejected because the current hosts already use ZFS datasets
  and tmpfs root would add workstation memory and disk-pressure tradeoffs.
- Recreate datasets during NixOS activation: rejected because disko owns initial
  dataset creation and activation-time destructive storage changes are risky.
- Always require reinstall: rejected because the current shared layout appears
  to contain the needed datasets and blank snapshot; existing hosts should be
  checked first.

## Decision: Activation requires a reviewed persistent system-state inventory

Before root reset can be enabled, the implementation must expose a reviewed
inventory of root-owned state that survives reset. The inventory uses simple
machine-readable path lists plus `systemState.reasons`, an attrset keyed by
exact live path or evaluated-state key. The minimum inventory covers:

- `/etc/machine-id` for stable machine identity.
- `/persist/etc/ssh/ssh_host_*` private and public host keys for stable SSH
  identity and sops host-recipient decryption. These keys are consumed directly
  by sops-nix and OpenSSH rather than by impermanence.
- ZFS identity/import state. In this repository `networking.hostId` is derived
  deterministically from `config.networking.hostName`, so validation must prove
  that evaluated value; if implementation introduces `/etc/hostid` or zpool
  cache files, those paths must be persisted or explicitly classified.
- Secret-decryption bootstrap dependencies. The user age key lives in
  `/run/secrets/<user>-age-key` and is regenerated by sops-nix, so it should
  not be persisted; the host SSH key that decrypts it must be stable.
- Host-conditional service state that would otherwise break enabled services,
  such as `/var/lib/tailscale` when Tailscale is enabled, and systemd timer
  state required by persistent timers.

Rationale: A clean root is only safe when the identity and bootstrap material
needed after boot are either declarative, regenerated from encrypted secrets, or
persisted. Missing SSH host keys would break sops host-recipient decryption;
missing machine identity or ZFS import state can change system identity or
prevent imports; missing service state can force unnecessary re-enrollment.

Alternatives considered:

- Keep the generic requirement to preserve "system state": rejected because it
  is too broad to drive task generation and too easy to miss boot-critical
  paths.
- Store reasons only in comments next to path lists: rejected because comments
  are not machine-checkable and cannot support assertions that every persisted
  path was reviewed.
- Use submodules for every persisted file and directory: viable, but rejected
  for this plan because upstream impermanence consumes path lists directly; a
  separate reasons attrset keeps transformation simple while preserving review
  metadata.
- Persist all of `/etc` and `/var/lib`: rejected because it would preserve too
  much undeclared drift and undermine the clean-root goal.
- Only rely on declarative regeneration: rejected because SSH host identity and
  selected service enrollment state are not safely regenerated on every boot.

## Decision: Root rollback is an initrd-stage operation, not activation work

The root reset implementation belongs in `modules/nixos/storage/impermanence.nix`
as an initrd-stage rollback hook. It must run after the encrypted ZFS pool is
available and before the root filesystem is used as the live system root. It
must fail visibly if the configured dataset or blank snapshot is missing.

Rationale: A normal activation script or late systemd unit is too late: by then
the root filesystem may already contain mutable state for the current boot.
NixOS exposes initrd hooks such as `boot.initrd.postDeviceCommands` and
`boot.initrd.postMountCommands`; implementation must choose the hook or initrd
systemd unit that runs before `/` is used and validate the generated ordering in
the VM reboot test.

Alternatives considered:

- NixOS activation script: rejected because it runs against an already booted
  system and cannot safely reset the live root.
- Late multi-user systemd service: rejected because services may already have
  written state to the root dataset.
- Disko post-create hook only: rejected because it creates the blank snapshot
  but does not reset root on each boot.

## Decision: Keep `/home/<user>` persistent and add `/persist/home/<user>` safe-haven

The existing `/home` dataset remains persistent. Curated user safe-haven data is
stored under `/persist/home/<user>` and exposed at the expected
`/home/<user>/...` paths. Restic targets `/persist/home/<user>` for curated home
data; ordinary persistent home files remain available but are outside the
default backup source.

Rationale: The clarified spec explicitly keeps `/home/<user>` persistent for the
foreseeable future. The safe-haven path provides an auditable backup source
without turning ordinary home files into ephemeral data. `/persist` already
exists in the shared disko layout and is separate from the live home dataset.

Alternatives considered:

- Make `/home` ephemeral: rejected because it contradicts the clarified spec.
- Put the safe-haven under `/home/<user>/persist`: rejected because backup
  review is easier when the curated source lives outside the full home tree.
- Synchronize between `/home` and `/persist`: rejected because it adds
  conflict, deletion, and ordering failure modes that bind/link persistence
  avoids.

## Decision: Narrow restic backup sources, preserve targets and credentials

Restic backup paths will stop using the full live `/home` directory for affected
hosts and will include `/persist/home/<user>`. System-state backup inclusion is
explicit through `nixconfig.storage.impermanence.systemState.backupPaths`;
persisting a path for boot safety does not automatically add it to backups.
Existing restic repositories, schedules, retention, and sops-backed credentials
remain unchanged.

Rationale: The feature's storage value is making backup scope explicit instead
of maintaining an expanding exclude list. Existing backup destinations and
credentials already work and are outside the requested change.

Alternatives considered:

- Continue backing up `/home` with excludes: rejected because it is the problem
  the feature is meant to remove.
- Back up all of `/persist`: rejected because system state, user data,
  containers, VM disks, and logs have different backup needs.
- Automatically back up every persisted system-state path: rejected because
  boot-critical state and backup-worthy state are overlapping but not identical,
  especially for sensitive service enrollment keys.
- Create new backup repositories: rejected because the spec requires retaining
  existing targets and credentials.

## Decision: Validate with narrow builds plus VM or host reboot tests

The implementation will use narrow Nix builds for `hephaestus-vm`, `hephaestus`,
`apollo`, and the standalone Home Manager profile, then validate runtime
behavior through a VM or controlled host reboot before deploy.

Rationale: Build evaluation proves configuration wiring and option validity, but
impermanence behavior depends on boot ordering, mounts, root rollback, and
service startup. Runtime reboot validation is required to prove root wipe,
system-state persistence, safe-haven exposure, and backup source narrowing.

Alternatives considered:

- Only run `nix flake check`: rejected as too broad for iteration and
  insufficient for reboot behavior.
- Only inspect generated configuration: rejected because mount and rollback
  ordering needs runtime validation.
